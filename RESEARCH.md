# 今日热搜 · 需求文档（优化版）

> **更新日期**：2026-05-29
>
> **项目定位**：一个轻量级的多平台热搜聚合网站，用户可通过链接访问，首页以卡片网格形式展示各平台实时热榜。

---

## 一、调研发现

- 现有方案（如"今日热榜"）采用"平台卡片"布局，每张卡片对应一个站点的榜单
- 每条热搜至少包含：排名、标题、原文链接，可选展示热度值
- 因浏览器跨域限制及反爬策略，前端无法直接抓取第三方热榜，需后端中转

---

## 二、核心需求

### 2.1 平台要求（≥ 3 个）

| 平台 | 说明 |
|------|------|
| 微博热搜 | 实时榜单 |
| 知乎热榜 | 热门话题 |
| B站热门 | 视频热度 |

> 卡片设计：**一个平台 = 一张独立卡片**

### 2.2 数据字段（每条热榜）

| 字段 | 必选 | 说明 |
|------|:----:|------|
| rank | ✅ | 数字排名（1 起始） |
| title | ✅ | 标题文本 |
| url | ✅ | 原文链接（可跳转回原平台） |
| heat | 可选 | 热度 / 阅读量 / 讨论数（展示为"xxx 万热度"） |

### 2.3 数据真实性

- 必须对接**真实公开接口**或可解析页面
- **严禁**长期使用 Mock / 静态假数据（仅允许开发初期临时验证）

### 2.4 后端缓存策略

| 配置项 | 说明 |
|--------|------|
| 缓存有效期 | 5～10 分钟（可配置，**默认 5 分钟**） |
| 缓存范围 | 每个平台的热榜列表 |
| 目的 | 降低被上游限流风险，提升响应速度 |

### 2.5 部署要求

- 支持 **HTTPS 公网访问**
- 可部署于常见环境（VPS、云函数 + 对象存储、容器等）

---

## 三、风险与约束说明

| 类型 | 说明 |
|------|------|
| **上游稳定性** | 第三方热榜接口可能变更、返回结构变化或限制 IP |
| **反爬措施** | 部分平台需伪装 UA / 添加延迟，必要时可降级为公开 RSS 或非官方 API |
| **使用性质** | 仅作为个人 / 学习项目，页脚需明确标注"**非商用 · 数据来自第三方平台**" |

---

## 四、非功能性需求（建议）

| 需求 | 说明 |
|------|------|
| **响应式布局** | 支持 PC / 手机自适应 |
| **加载体验** | 支持骨架屏或局部刷新（避免整页重新加载） |
| **异常容错** | 单个平台抓取失败时，不影响其他卡片展示（展示"暂时无法获取"） |
| **可维护性** | 新增平台时改动量可控（建议统一 adapter 接口） |

---

## 五、可选扩展（不作强制）

- 支持手动刷新单个卡片（绕过缓存）
- 支持折叠 / 展开卡片
- 提供简单的 API 接口（如 `/api/hot?platform=weibo`）

---

## 六、开发里程碑（参考）

| 阶段 | 内容 |
|------|------|
| **Phase 1** | 技术选型、架构设计、adapter 接口定义 |
| **Phase 2** | 后端服务搭建 + 单平台数据对接（微博） |
| **Phase 3** | 多平台扩展 + 缓存策略实现 |
| **Phase 4** | 前端页面开发 + 响应式适配 |
| **Phase 5** | 部署上线 + 监控告警 |
